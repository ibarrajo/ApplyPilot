# Apply UX Overhaul — Design (2026-04-25)

> **Status:** Approved 2026-04-25. Replaces `2026-04-25-apply-ux-overhaul-WIP.md` (deleted in same commit). Implementation plan generated by writing-plans next.

## 1. Summary

The Tier-3 apply layer (Claude Code subprocess driving Playwright via CDP) is re-architected around a **custom MV3 Chrome extension** running on **Chrome for Testing 148 + patchright**. The extension becomes the sole user-facing surface during paused / human-in-the-loop states, replacing the deleted CDP-injected always-on badge (commit `5860c1e`). Four near-duplicate HITL paths in `launcher.py` collapse into one helper. `launcher.py` (3,273 lines) decomposes into three modules. Per-worker file scoping fixes a race in resume copy. A new `--no-hitl` mode pauses-and-moves-on instead of pauses-and-waits.

## 2. Background

### 2.1 Why Chrome for Testing

Chrome 137 removed the `--load-extension` flag from all branded Chrome builds (Stable / Beta / Dev / Canary). The Chromium maintainers explicitly carved out **Chromium and Chrome for Testing** as the supported automation paths ([source](https://groups.google.com/a/chromium.org/g/chromium-extensions/c/1-g8EFx2BBY/m/S0ET5wPjCAAJ)). CfT is Google's officially-maintained automation Chrome — same engine as branded stable, ships in lockstep, no policy blocks. Custom extensions install cleanly via `--load-extension={path}`. Fallback is Ungoogled Chromium (also honors the flag).

### 2.2 Why patchright

`patchright` is a drop-in `playwright` Python replacement (3k+ stars, active April 2026) that strips standard automation fingerprints: `navigator.webdriver`, `--enable-automation`, `Runtime.enable` CDP leak. Pairs with `channel="chrome"` + persistent `user_data_dir` for real-Chrome cookie shape. The agent-side `playwright-mcp` continues to drive form interactions; patchright covers launch-time fingerprints. Net: meaningfully better than today's stock Playwright launch, not airtight (CDP-client-side stealth is partial).

## 3. Architecture

### 3.1 Runtime layers (per worker)

**Browser.** Chrome for Testing 148 binary at `~/.applypilot/chrome-for-testing/chrome-linux64/chrome`. CDP port `9222 + worker_id`. Per-worker user-data-dir `apply-workers/worker-{wid}/`. Resume copy at `apply-workers/worker-{wid}/resume.{pdf,docx}` (per-worker scope, fixes audit #9).

**Extension** (MV3, custom, loaded via `--load-extension`):

- **Service worker** (`background.js`): per-job tab tracker (`tabs.onCreated` + `openerTabId`); action-log ring buffer in `chrome.storage.local`; long-polls launcher's `/api/status`; relays popup actions out via `fetch`. `chrome.alarms` 60s heartbeat keeps SW alive during pauses.
- **Content script** (`content.js`, in-set tabs only): filtered event capture (button-target clicks, anchor navs, form submits, tab opens — not body/scroll/focus); on Resume snapshots `<input>/<select>/<textarea>` values; renders banner when SW signals paused.
- **Popup** (`popup.html` + `popup.js`): liveness LED, current job title, four action buttons (Mark Applied / Skip / Take Over / Resume), HITL toggle indicator.

Per-install random extension `key` (manifest field) so the extension ID is unique per ApplyPilot install, defeating LinkedIn's bulk fingerprinting of `web_accessible_resources` across users.

**Launcher** (Python). HTTP server on port `7380 + worker_id` (existing). CDP attach. Agent subprocess spawn. State-machine transitions. `launcher.py` (3,273 lines) decomposes:

| New module | Approx LOC | Contents |
|---|---|---|
| `apply/orchestrator.py` | ~800 | `worker_loop`, `main`, `_worker_loop_body` |
| `apply/result_handlers.py` | ~600 | `mark_*`, `reset_*`, `release_lock`, run-job result parsing |
| `apply/hitl.py` | ~300 | `_run_hitl(worker_id, port, job, reason, instructions, navigate_url)` |
| `apply/chrome.py` (existing, simplified) | smaller | CfT binary + patchright launch; `setup_worker_profile` simplified |

Standalone `applypilot human-review` (port 7373) **deleted** — launcher-restart path (§4.3) covers its only remaining use case.

**Communication topology:**
- Extension → launcher: HTTP `fetch` to `localhost:7380+wid`
- Launcher → extension: shared state served on `/api/status`; extension long-polls

No CDP-side script injection.

### 3.2 Per-job tab tracking

Extension binds to a worker_id at launch via initial `chrome.storage.local` write during `setup_worker_profile` (launcher-side, before Chrome starts). The worker's "primary tab" is the first tab loaded under that worker. Any tab opened from the primary or its descendants — tracked via `openerTabId` — joins the job's in-set. Tabs the user opens manually (no opener) are out-of-set.

In-set tabs:
- Receive content-script injection
- Contribute to action log
- Render banner when paused

Out-of-set tabs are ignored. Handles OAuth popups, verification windows, and side-tabs without classifying them by role.

### 3.3 HITL collapse

Single helper `_run_hitl(worker_id, port, job, reason, instructions, navigate_url)` replaces:
- Generic HITL path (`launcher.py:2725-2816`)
- screening_questions path (`launcher.py:2640-2723`)
- login_required path (`launcher.py:2820-2895`)
- post-Chrome-crash recovery (`launcher.py:2779-2789`, `2870-2890`)

Estimated ~150 lines deleted.

Common steps:

1. `transition_state(job.url, needs_human, reason)` — atomic state + audit row.
2. Update `/api/status` shared state to `{paused: true, reason, instructions, navigate_url}`. Extension picks up via long-poll.
3. Wait on `done_event` (set by `/api/done/{hash}` POST or terminal stdin fallback).
4. Done POST body includes the log: `{events: [...], snapshots: {tab_url: {field: value, ...}}}`. (Single round trip — no separate `/api/action-log` call.)
5. Build agent prompt section from log (§4.4).
6. `transition_state(job.url, applying, "hitl_resume")`.
7. Re-spawn `claude` subprocess on same Chrome (CDP still attached).
8. Return result.

## 4. Data flow

### 4.1 Pause cycle (HITL on, default)

```
Agent emits RESULT:NEEDS_HUMAN:{reason}:{url}
  └─ Launcher
     ├─ transition_state(needs_human, reason)
     ├─ Persist needs_human_reason / needs_human_url / needs_human_instructions
     └─ _run_hitl(...)
        ├─ Update /api/status: {paused: true, reason, instructions, navigate_url}
        ├─ Extension SW (long-poll) picks up
        ├─ Content script renders banner in in-set tabs
        ├─ User completes form (events captured to ring buffer in storage.local)
        ├─ User clicks Resume in popup
        │   └─ SW asks content scripts for form snapshots
        │   └─ SW POSTs /api/done/{hash} with {events, snapshots}
        ├─ Build USER ACTIONS DURING PAUSE prompt section from log
        ├─ transition_state(applying, "hitl_resume")
        └─ Re-spawn claude on same Chrome → continues
```

### 4.2 `--no-hitl` path

Steps 1-2 unchanged. Step 3 just releases the lock and returns (no `_run_hitl` invocation, no banner, no waiting). Worker pulls next job. The job stays in `needs_human` state; user can later run `applypilot apply` (HITL on) and the launcher's "Chrome dead" restart path picks it up.

### 4.3 Launcher restart paths

**Chrome alive** (e.g., user Ctrl+C'd launcher mid-pause, Chrome still running):
- New launcher probes CDP port `9222 + wid`. If responsive: re-attach; read pause state from extension via CDP `Runtime.evaluate("chrome.storage.local.get(...)")`; if SW reports paused, resume the wait loop without re-displaying banner.

**Chrome dead** (Chrome crashed, machine rebooted, etc.):
- Launcher queries `state = needs_human` jobs. Spawns fresh CfT instance per worker. Navigates to `needs_human_url` from DB. Sets `/api/status` to `paused`. Extension content script renders banner with stored instructions on first paint. User completes, same Resume path.

### 4.4 Action log shape (in agent prompt on Resume)

```
USER ACTIONS DURING PAUSE:
Timeline (primary tab — workday.com/.../step-2):
  +0:12  Navigated /login → /sso
  +0:18  Clicked "Sign in with LinkedIn"
  +0:22  Tab opened: linkedin.com/oauth/authorize
  +0:34  (in linkedin tab) Clicked "Allow"
  +0:36  Tab closed; primary tab now /step-2
  +0:48  Filled custom-q-1, custom-q-2
Form values now in tabs:
  workday.com/.../step-2:
    email: alex@example.com
    phone: 206-555-...
    custom-q-1: "5+ years"
    custom-q-2: "Yes"
Final URL per in-set tab:
  workday.com/.../step-2 (primary)
```

Buffer caps: 200 events / 64KB total per pause. Excess oldest-evicted with `[N events truncated]` marker.

## 5. Error handling

| Failure | Behavior |
|---|---|
| SW idle-killed mid-pause | `chrome.alarms` 60s heartbeat respawns it; pause state in `chrome.storage.local` survives. Content script reconnects on next `runtime.connect`. |
| Chrome crashes mid-pause | Launcher's CDP disconnect triggers `release_lock` (already shipped, `a3f332f`). Next run re-launches per "Chrome dead" path (§4.3). |
| Resume button broken (extension fault) | `_run_hitl` exposes terminal fallback: prints `[hitl] paused on {url}; type 'done' / 'skip' / 'fail' to override` once at pause start, reads stdin in a background thread. |
| Smoke test A/B/C fails | Fall back to **Ungoogled Chromium**; design unchanged (CfT and Ungoogled both honor `--load-extension`). |
| LinkedIn extension fingerprint | Empty `web_accessible_resources` in manifest; per-install random `key` field. |
| Action log poisoned (huge paste) | Whitelist event types; cap 200 events / 64KB; oldest-evicted. |
| HITL toggle mid-pause | Persists to `chrome.storage.local`; takes effect on next pause. Current pause completes via Resume. |
| Skip race (user clicks Skip while agent acts) | `transition_state(needs_human → archived)` is atomic; agent's next emit hits stale-job error → launcher releases lock cleanly. |

## 7. Testing

- **Unit (~15 new tests):** `_run_hitl` happy path + each error branch (mock launcher state, mock extension responses); action-log parser (event/snapshot serialization round-trip); per-job tab classifier (opener-chain edge cases including same-origin popups and manual-user tabs); state-machine transitions for `needs_human → archived` (Skip) and `needs_human → applying` (Resume).
- **Integration:** `tests/test_hitl_flow.py` end-to-end with real CfT + extension, fake job served from local HTTP fixture. Three scenarios: HITL-on Resume, `--no-hitl` move-on, launcher-restart-Chrome-dead replay.
- **Manual smoke (first plan task, before any code change):** WIP §5 steps A/B/C — CfT 148 login on a Workday tenant + LinkedIn, patchright import swap on existing flow, drop current extension into CfT and confirm popup loads.
- **Regression:** existing 230 tests stay green. No DB migrations beyond optional columns; logs live in extension storage, no schema needed.

## 8. Out of scope / risks

**Out of scope:**
- Mark Applied → email follow-up integration (separate Tier-2 work in `tracking/`).
- Action-log replay debugging UI (post-MVP).
- CAPTCHA detection (still falls to HITL).
- Login persistence beyond worker user-data-dir (already covered by per-worker profiles).
- Tier-2 stages (discover/enrich/score/tailor/cover) — untouched.

**Live risks:**
- **MV3 SW lifetime.** Heartbeat alarm + `storage.local` cover most cases; Chrome may still kill SW under memory pressure; fallback is content-script reconnect via `runtime.connect`.
- **patchright × playwright-mcp interop.** Launcher-side fingerprints handled; CDP-client-side stealth is partial since the agent's playwright-mcp is a separate CDP client.
- **LinkedIn extension fingerprinting.** Smoke phase tests this; if randomized ID isn't enough, fallback is to remove unique-named files in extension.
- **Ungoogled Chromium fallback drift.** Lags upstream by 1-2 weeks; "no Google pings" is itself a fingerprint signal. Acceptable downside if CfT smoke fails.
